Executive Summary

This project proposes a modular, low-power communication mesh designed for early-stage and long-term Mars exploration missions. The system uses a combination of 1U CubeSat relay nodes and ground-based emergency stations to provide resilient, self-healing communication between astronauts, rovers, and surface habitats.

Unlike traditional point-to-point or centralized relay systems, this approach emphasizes distributed redundancy, allowing communication to persist even when individual nodes fail. The architecture prioritizes rapid deployment, energy efficiency, and scalability, making it suitable for harsh, infrastructure-poor environments such as Mars.

The Problem

Mars terrain creates severe line-of-sight and signal attenuation challenges due to dust storms, elevation changes, and obstacles.
Fixed communication infrastructure is slow, expensive, and risky to deploy during early missions.
Communication failures during emergencies can result in loss of life, mission failure, or stranded crews.
There is currently no lightweight, rapidly deployable solution for inter-settlement connectivity on Mars.

System Architecture

The Mars Communication Mesh is composed of three tightly integrated layers: orbital relay nodes, a wireless surface mesh, and intelligent ground stations. Together, these components form a fault-tolerant network capable of adapting to node loss and environmental changes.

1U CubeSat Node

Acts as an orbital or elevated relay node. Each CubeSat carries an ESP8266-based controller, Zigbee radio, and attitude control hardware for antenna alignment and mesh optimization.

Wireless Mesh Layer

A Zigbee-based mesh network enables multi-hop routing, allowing data to dynamically reroute around failed or obstructed nodes without human intervention.

Ground Base Stations

Fixed or semi-mobile stations that provide emergency beacons, sensor data, and communication access points for astronauts and rovers.

Telemetry & Communication Analysis

Reliable telemetry is essential for sustaining communication between CubeSat relay nodes, surface base stations, and mobile assets. The system utilizes a PCB-integrated microstrip patch antenna designed to provide directional radiation, compact form factor, and efficient power utilization.

Patch Antenna Design & Theory

A microstrip patch antenna consists of a conductive radiating patch separated from a ground plane by a dielectric substrate. Radiation occurs due to fringing electric fields at the patch edges when the antenna resonates at its design frequency.

This antenna type is well suited for CubeSat applications due to its low profile, lightweight structure, and direct PCB integration, eliminating the need for external antenna assemblies.

Figure 11: PCB-integrated microstrip patch antenna designed for the Zigbee communication band, showing radiating patch geometry and ground plane layout.

Gain & Radiation Characteristics

The patch antenna exhibits a directional radiation pattern, concentrating transmitted energy normal to the patch surface. This improves link efficiency compared to omnidirectional antennas, particularly in power-limited space systems.

Antenna gain is achieved through controlled directivity rather than active amplification. The measured gain provides sufficient link margin while maintaining compliance with power and size constraints.

Link Budget Computation Code

The following code snippet demonstrates the methodology used to compute received power and link margin for the communication system: 


nealinkb.mlx:
clc; clearvars;

f = 433e6; % frequency in Hertz
r = 450e3; %altitude (change later, based on elevation angle) in metres

P_t = 15; % includes HPA, Power of the transmitter in dBw
G_t = 2; % gain of the antenna, in dBi 
L_txsystem = 0; %losses due to feeding lines, connectors, filters, HPA etc in dB
L_fspl = 0; %free space path loss (calculation below) in dB
L_atm = 0; %atmospheric, rain, clouds, fog, ionosphere attenuation + (Diffusion- due to obstacles- neglect) + (Multipath fading- only for terrestrial- neglect) in dB
L_plrz = 0; %polarization mismatch, effectively 0 for circularly polarised antennas in dB
L_point = 1.5; % pointing losses, add both reciever and transmitter pointing losses in dB

G_r = 3; % gain of ground station in dBi
G_lna_ext = 0; %gain of external lna, see below for effect on F in dBi
L_rxsystem = 1; %losses due to feeding lines, connectors, filters, LNA, demodulators etc in dB

S_datasheet = -135; %sensitivity on datasheet (already includes the internal LNA gain effect an noise spectrum density change) in dBm
T_ant = 750; %Thermal temperature, Tsky(atmospheric thermal temp) + Tgnd(due to sie and back lobes) in K
B_datasheet = 125e3; %Bandwidth of reciever in Hz
SNRrequired_datasheet = -17.5; %required signal to noise ratio for the reciever to be able to demod the signal in dB

U_bitrate = 980e-6; %bitrate that WE pass, megabits per second
EbNo_threshold = 7.1; %minimum Eb/No required by the reciever, in dB

EIRP = P_t + G_t - L_txsystem %effective output power in dBm
L_fspl = 20*log10(4*pi*r*f/299792458)
L_prop = L_fspl + L_atm %propogation losses

Pr = EIRP - L_prop - L_plrz - L_point + G_r + G_lna_ext - L_rxsystem;
C = Pr %recieved power to digital system at reciever end, in dBm

N = S_datasheet - SNRrequired_datasheet %noise power
NF = N + 174 - 10*log10(B_datasheet) %noise figure at datasheet
F = 10^(NF/10) %noise factor at datasheet

% if external lna connected - F = Flna + (Frx - 1)/G_lna_ext;

Tr = (F-1)*290 %Thermal temp at reciever end
Ts_actual = T_ant + Tr %Thermal temp of system incluing Tatm and Tr (entire system)
Ts_spec = F*290 %Thermal temp use by mnufacturer to calculate
correction = 10*log10(Ts_actual/Ts_spec) %change due to Tatm

S_actual = S_datasheet + correction; %actual sensitivity of reciever system
Psensitivity = S_actual

LINKMARGINP_via_sensitivity = Pr-Psensitivity

No = 1.380649e-23 * Ts_actual; %Noise spectral density in W/Hz
No = 10*log10(No) %nsd in dBW/Hz

C_ = C - 30; %C in dBW
CNo = C_ - No %received carrier power/noise spectral density in dB-Hz
EbNo_calculated = CNo - 10*log10(U_bitrate) -60 %received EbNo in dB

LINKMARGINP_via_ebno = EbNo_calculated-EbNo_threshold

Local Telemetry & Control Dashboard

This dashboard provides a real-time mission-control interface for monitoring system health, environmental conditions, and CubeSat orientation on Mars. The visual design emphasizes high contrast and clarity to support rapid decision-making during both nominal operations and emergency events.

Telemetry from the ground station and CubeSat nodes is transmitted to a locally hosted server using the MQTT protocol. The dashboard subscribes to these streams to render orientation data, sensor values, alerts, and communication status with minimal latency.

Figure 8: Mission-control dashboard displaying real-time CubeSat orientation, environmental telemetry, system alerts, and communication status.

A central 3D visualization displays the CubeSat attitude in real time using IMU data from the onboard MPU6050. Orientation updates are streamed at approximately 20 Hz, enabling smooth visualization of pitch, roll, and yaw. The model rotates about its geometric center to maintain a stable reference frame.

Environmental telemetry from the ground station, including atmospheric temperature and solar luminosity, is updated at 1 Hz and displayed as time-series plots with fixed axis scaling to preserve contextual accuracy.

Persistent alert panels visually latch when abnormal conditions are detected, including seismic activity and reduced atmospheric clarity due to dust storms. A rolling event log records system messages and emergency signals with timestamps, while uplink indicators confirm active communication with CubeSat relay nodes.

// MQTT Telemetry Publisher (ESP8266)
void publishTelemetry() {
  StaticJsonDocument<256> data;
  data["temp"]  = temperature;
  data["lux"]   = luminosity;
  data["yaw"]   = imuYaw;
  data["pitch"] = imuPitch;
  data["roll"]  = imuRoll;

  char buffer[256];
  serializeJson(data, buffer);
  mqttClient.publish("mars/telemetry", buffer);
}

// IMU Orientation Processing (MPU6050)
void computeOrientation() {
  imu.getMotion6(&ax, &ay, &az, &gx, &gy, &gz);
  pitch = atan2(ax, sqrt(ay*ay + az*az)) * RAD_TO_DEG;
  roll  = atan2(ay, sqrt(ax*ax + az*az)) * RAD_TO_DEG;
  yaw  += gz * deltaTime;
}

Impact & Vision

The Mars Communication Mesh enables reliable emergency response, resource coordination, and continuous connectivity between humans and machines. Beyond technical utility, the system supports human morale and safety by ensuring astronauts are never completely isolated.

In the long term, this mesh network can serve as a foundational layer for Martian settlements, expanding organically as new nodes are deployed, ultimately forming a resilient planetary communication backbone.
